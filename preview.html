Executive Summary

00｜结论先行：这个项目有必要做，但必须按“能力资产基础设施”而不是“新 Skill 平台”建设

当前集团已经具备云盘与知识、Wiscode和纳米work、Agent 协作、MCP 与内部工具、Skill 市场分发等多项基础能力。 真正缺失的不是又一个 Agent 客户端，也不是又一个 Skill 市场，而是把个人在真实工作中形成的方法、经验、流程和工具组合， 持续沉淀为团队与组织可复用能力的基础设施。

建议决策

建议立项验证 SkillOps，但第一阶段只做轻量嵌入式能力：查重、创建、质检、Fork、同步、使用反馈和改进建议； 不重新做 Skill 市场，不要求用户迁移到新平台，不在需求未验证前建设大而全的治理系统。

1

必要性

Skill 已经大量自然产生，但缺少生产、复用、版本、反馈与持续进化机制，重复建设和经验流失正在发生。

2

产品价值

让员工在原 Agent 客户端中直接把一次高质量任务沉淀成 Skill，并在写之前查重、写的时候质检、使用后持续迭代。

3

战略价值

补齐知识、任务、协作、工具和复盘之间的能力资产层，支撑超级个体、超级团队和超级组织。

4

长期结果

企业不只积累知识和 Agent，更能积累可测试、可版本化、可共享、可治理、可持续进化的 AI 能力资产。

5

落地形态

SkillOps Core + MCP Server + Agent 客户端轻入口 + 后台治理台；普通用户不进入独立平台完成主流程。

6

协同关系

云盘提供知识，Agent 引擎和 Loop 完成任务，Skill 市场审核分发，SkillOps 负责能力生产、治理和进化。

项目的一句话定位

SkillOps 是面向企业 Agent 与 Loop 体系的能力资产生产、治理和持续进化基础设施。它让能力从超级个人的智能放大，沉淀为超级团队的协同资产，并进一步驱动超级组织的系统进化。

Why This Matters

01｜为什么做：必要性、内部现状、核心矛盾与产品边界

Why Now

1.1｜项目必要性：企业 AI 基建正在从“知识管理”走向“能力资产管理”

1. 从认知共识走向能力资产化落地

知识决定企业之间的差异，这个判断是整个项目最重要的战略起点。但知识本身并不直接等于生产力： 知识只有被 Agent 在真实任务中调用，被流程验证，被团队复用，并通过复盘持续修正，才会形成企业的生产力和差异化能力。

云盘和知识库解决“知识在哪里、如何被找到”；SkillOps 解决“知识如何变成 Agent 可执行、可复用、可持续进化的工作方法”。

3. Agent 引擎六要素中，SkillOps 补的是能力进化链

要素集团已有基础SkillOps 补充的能力
目标超级个体、超级团队、超级组织的目标体系把目标相关的高频方法沉淀成可复用能力。
任务Wiscode和纳米work把一次任务中验证有效的过程保存为 Skill，并持续优化。
协作Agent 协作平台支持团队共享、Fork、版本血缘、贡献回源和能力共创。
工具MCP、API、内部业务系统声明 Skill 的工具依赖、权限边界、平台适配和失败处理。
知识云盘与知识库改造把文档、规范、经验和案例转化为 Agent 可执行的方法。
复盘任务过程与使用数据从失败、人工修改和使用习惯中提取 Skill 改进信号。

4. 如果缺少这一层，企业会持续发生什么

  • 每个团队在不同客户端里独立写 Skill，方法无法共享。
  • A 已经解决的问题，B 和 C 仍然重复建设。
  • 一个高质量任务完成后，经验停留在聊天记录或个人本地。
  • 多个 Loop 在重复试错，但结果不能沉淀成组织能力。
  • Skill 市场有内容，却缺少质量、复用、版本和持续进化依据。
  • 企业拥有越来越多 AI 工具，却难以形成区别于其他企业的能力资产。
Current State

1.2｜内部现状与核心矛盾：有分发平台，但没有能力资产治理层

1. 当前已经具备的基础

能力方向当前基础
知识云盘业务正在进行 AI 化改造。
任务已有Wiscode和纳米work承载 AI 任务执行。
协作已有类似飞书的 Agent 协作平台。
工具MCP 与内部工具调用能力正在接入。
分发已有内部 Skill 市场，承担上架、安全审核、分发和安装。

2. 当前真实问题不是“不会写”，而是“无法从个人能力升级为组织能力”

内部现象背后的结构问题
各团队自己写自己的 Skill缺少统一的创建、描述和质量标准。
A 写了 Skill，B 不知道缺少创建前的相似 Skill 搜索和复用推荐。
B 基于 A 改了，但只在本地使用缺少 Fork、版本血缘和贡献回源机制。
多人重复创造类似 Skill缺少跨个人、团队和市场的查重机制。
Skill 上架只做安全审核缺少质量、测试、适配、复用和维护状态检查。
业务 Skill 很单点、即用即抛缺少临时、个人、团队、组织四类生命周期分层。
Skill 使用后没有反馈沉淀无法知道哪个版本更好、哪里经常失败、用户为何反复修改。

3. 当前主要矛盾

当前主要矛盾不是企业未来需不需要 Skill，而是能否把自然生长的个人 Skill，以足够低的摩擦转化为可复用、可治理、可持续进化的团队与组织能力资产。

4. 第一阶段最需要证明的不是“能生成”，而是“能改变行为和结果”

第一阶段必须验证：

  • 用户是否会在创建前复用已有 Skill，而不是继续从零开始。
  • 用户是否愿意在修改后提交回源，而不是只保留本地版本。
  • 质量检查是否能减少返工、提高任务成功率和输出稳定性。
  • 团队负责人和 Skill 市场是否愿意采用质量与版本数据。
  • 高价值 Skill 是否能从个人版本自然升级为团队和组织标准。
Product Positioning

1.3｜产品定位、边界与 Skill 分层

1. 产品定位

SkillOps 不是一个新的 Skill 市场，也不是单纯的 Skill 生成器，而是嵌入现有 Agent 客户端和企业系统的能力生产、质量、复用、版本与进化层。

2. SkillOps 与现有 Skill 市场的本质区别

系统主要职责核心价值
SkillOps创建、查重、质检、测试、Fork、版本、使用反馈、持续改进管理 Skill 从产生到进化的完整生命周期。
内部 Skill 市场正式上架、安全审核、组织分发、安装、下架管理组织级 Skill 的正式发布和分发状态。

3. 四类 Skill 生命周期

类型典型特征是否必须上架对应产品策略
临时 Skill一次性任务、即用即抛否快速生成、本地使用、无需重治理，可自动丢弃。
个人 Skill个人反复使用不一定个人库、版本记录、可分享、可升级为团队版。

组织 Skill跨团队复用、正式推荐是安全审核、质量门禁、适配检查、市场分发与持续维护。

5. 当前明确不做什么

不做

  • 不重新建设一个 Skill 市场。
  • 不替代现有 Agent 平台或 Agent 引擎。
  • 不强迫用户进入独立 Web 平台写 Skill。
  • 不把所有一次性 Skill 都纳入重治理。
  • 不在第一阶段做复杂审批和全平台深度适配。
  • 不把产品做成通用 Agent OS。

只做

  • 写之前查重与复用推荐。
  • 写的时候生成、质检和测试。
  • 改的时候记录 Fork 与版本血缘。
  • 用之后采集元数据并推送改进建议。
  • 达到组织标准后连接现有 Skill 市场。
  • 让能力能够跨人员、团队和客户端复用。
User Journey & Product Value

02｜怎么用：一个 Skill 的完整生命周期与产品价值落地

End-to-End Scenario

2.1｜完整用户场景：一个 FDE Skill 如何从个人经验变成组织资产

下面用集团内部 FDE / 交付工程师场景，展示产品如何进入用户真实工作、如何产生价值，以及如何与现有系统形成完整周期。

阶段 1｜真实任务中产生经验

FDE 小王正在为客户制作“普惠金融智能体”方案。他在 WorkBuddy 或类 Codex 产品里，反复让 Agent 读取客户资料、提炼业务痛点、 生成场景、补充交付边界并形成评审材料。经过多轮修改后，已经形成一套稳定方法。

过去，这套方法可能停留在聊天记录、个人 Prompt 或本地 Skill 中；其他同事不知道，下一次项目仍然从零开始。

产品介入：任务结束后提示“这个流程已多次复用，是否沉淀为 Skill？”用户可保存为个人 Skill、团队 Skill或暂不保存。
阶段 2｜自动抽取 Skill 草稿

用户点击“保存为 Skill”后，不需要离开客户端或重新写一遍。系统从当前任务中抽取 Skill 名称、适用场景、输入材料、执行步骤、 输出格式、不适用边界、工具依赖、参考资料和测试样例。

示例草稿：
Skill 名称：普惠金融智能体方案生成 Skill
输入材料：客户背景、业务流程、访谈纪要、产品能力说明
输出结果：场景清单、方案结构、交付边界、客户评审材料初稿
不适用场景：不直接生成最终合同、报价或法律承诺
依赖能力：云盘资料检索、企业知识库、文档生成、Agent 模板库
产品价值：业务和交付人员无需理解复杂 Skill 格式，也能把真实工作方法沉淀下来。
阶段 3｜创建前自动查重

系统先查询个人库、团队库和已同步的市场目录，发现已有相似 Skill：

相似 Skill相似度所属团队建议
智能体方案初稿生成 Skill82%FDE 交付团队建议 Fork 修改
客户访谈纪要转方案 Skill68%解决方案团队可参考
投标材料智能体配置 Skill55%政企交付团队可复用部分规则
产品价值：把“个人重复创造”变成“组织已有能力复用”。
阶段 4｜Fork 与版本血缘

小王基于已有 Skill 创建业务分支，系统记录原始来源、修改者、修改内容、适用业务、版本和是否建议回源。 本次新增普惠金融业务字段、风险提示和客户评审材料格式。

产品价值：允许个人灵活修改，同时避免改进只留在本地；一个版本的变化可以被组织看见。
阶段 5｜质量检查与测试补齐

保存前，SkillOps 对场景、输入、输出、边界、工具依赖、测试样例和平台适配进行检查。

检查项结果
场景是否明确通过
输入材料是否明确通过
输出格式是否明确部分缺失
不适用边界是否明确缺失
是否有测试样例缺失
是否声明工具依赖通过
是否适合直接上架暂不建议

用户可一键补充正常样例、边界样例、输出质量标准和不适用场景。

产品价值：让“作者自己能用”升级为“其他人也能稳定使用”。
阶段 6｜团队成员复用

一周后，小李要为能源行业生成类似方案。系统在他创建前推荐已有 Skill，小李选择基于小王版本 Fork，并补充能源行业规则。

产品价值：单人效率提升开始转化为团队能力共创。
阶段 7｜合并与团队标准版

当普惠金融、能源、政务、港口等多个分支出现后，系统提示团队负责人：多个版本来自同一主 Skill， 建议合并公共部分，把行业差异作为参数或参考资料保留。

负责人可选择合并为团队标准版、保留业务分支、指定主版本或申请上架。

产品价值：把自然生长的个人能力整理为团队标准能力。
阶段 8｜进入 Skill 市场上架前检查

团队版 Skill 准备进入内部市场时，SkillOps 生成质量、相似度、测试、工具依赖、适配范围、复用记录和维护责任人报告； 现有 Skill 市场继续承担安全审核和正式分发。

示例：该 Skill 已在 FDE 团队内使用 38 次，被 6 人 Fork，覆盖 4 类行业场景；质量评分 86 分，测试样例完整， 建议作为集团推荐 Skill 上架，适用 FDE、解决方案和售前团队。
产品价值：市场不再只有“安全审核 + 分发”，而是拥有质量与复用依据。
阶段 9｜上线后持续监控与改进

SkillOps 记录安装与使用次数、使用客户端、触发方式、执行成功、人工修改、Fork、回源、评分和版本活跃度等元数据。 默认不采集完整 Prompt、输出全文和业务敏感文件。

示例建议：该 Skill 最近 30 天被使用 126 次，其中 42 次用户手动修改了输出格式， 建议将“客户评审材料格式”升级为默认输出模板；能源行业的修改比例较高，建议保留独立行业扩展版。
产品价值：Skill 不再是一次性文件，而是根据真实使用持续迭代的能力资产。

Product Value

2.2｜产品价值如何落地：不同角色分别得到什么

1. 对一线员工：无迁移成本地沉淀和复用自己的能力

员工仍然在 Codex、Claude、WorkBuddy 或Wiscode和纳米work里工作，只需要使用几个自然入口：

查找类似 Skill 保存为 Skill 检查当前 Skill Fork 一个 Skill 提交团队共享 查看改进建议

员工获得的直接价值：

  • 不用反复复制 Prompt，也不用每次从零开始。
  • 不需要理解复杂文件结构，即可把真实任务沉淀为 Skill。
  • 写之前能发现已有能力，避免重复创造。
  • 使用后能根据真实习惯持续优化自己的 Skill。

2. 对团队负责人：把个人经验变成团队标准

团队负责人可以看到：

  • 团队有哪些高频 Skill，哪些能力正在自然形成。
  • 哪些 Skill 被多人重复创建，哪些版本应该合并。
  • 哪些 Skill 质量低、失效或长期无人维护。
  • 哪些 Skill 已经被真实复用，可以升级为团队标准。
  • 哪些分支适合保留为行业或业务版本。
团队价值不是多一个管理看板，而是减少重复建设、提高新人上手速度、降低对少数专家个人经验的依赖。

3. 对 Skill 市场运营方：从“货架”升级为能力资产市场

SkillOps 为现有市场提供上架前和上架后的数据能力：

上架前

  • 质量检查报告
  • 相似 Skill 检查
  • 测试样例完整度
  • 工具依赖与平台适配
  • 维护责任人和复用证据

上架后

  • 使用与版本反馈
  • 失效和低活跃提醒
  • Fork 合并建议
  • 高价值 Skill 推荐
  • 需要升级或下架的依据

4. 对集团管理层：看见 AI 能力资产的形成过程

管理层不再只看到“多少人用了 AI、调用了多少次模型”，而可以看到：

  • 哪些业务知识真正转化为 Agent 可执行能力。
  • 哪些团队持续贡献高价值 Skill。
  • 哪些能力已经跨团队复用，哪些领域仍存在能力缺口。
  • 哪些内部能力具备对外 SaaS 或私有化产品化潜力。
Strategic & Synergy Value

03｜为什么重要：战略价值与现有 AI 基建协同

Strategic Value

3.1｜战略价值：补齐超级个体、超级团队、超级组织的能力资产底座

1. 支撑超级个体：从“会用 AI”到“拥有可进化的个人能力库”

超级个体不只是能用 Agent 完成一次任务，而是具备三种能力：

  1. 能用 Agent 完成工作。
  2. 能把自己的方法沉淀为 Skill。
  3. 能根据实际使用持续迭代自己的能力资产。

一个优秀 FDE 可以逐步沉淀客户访谈、方案生成、交付检查、智能体配置和评审材料等 Skill； 个人能力不再只存在于脑中，而是形成可积累、可迁移、可复用的能力库。

2. 支撑超级团队：从“每个人都用 AI”到“团队共同拥有和进化能力”

超级团队的关键不是每个人各自使用 AI，而是：

  • 个人能力能够被团队发现和复用。
  • 多人修改能够保留血缘并回到主版本。
  • 高价值实践能够沉淀为团队标准。
  • 新人可以直接调用成熟能力，而不是完全依赖师傅带教。

3. 支撑超级组织：从“拥有很多 Agent”到“能够管理组织的 AI 能力资产”

超级组织需要具备：

  • 看见组织内有哪些能力。
  • 知道哪些能力被高频使用、哪些正在失效。
  • 推动高价值能力跨团队复用。
  • 把复盘和运行反馈转化为能力更新。
  • 识别可以对外产品化的内部能力。

4. 对集团 AI 基建的长期意义

通用模型、通用 Agent 引擎和通用 Loop 会逐渐成为各家都能获得的基础能力；真正拉开企业差距的，是企业能否把自己的知识、流程、标准和实践反馈沉淀成持续进化的 Skill 资产。
Synergy Value

3.2｜协同价值：不替代现有系统，而是把它们连接成能力闭环

现有系统当前承担的能力SkillOps 的协同方式

SkillOps 不是新入口，而是连接知识、任务、协作、工具、市场和复盘的能力治理层。
Product Form & Evolution

04｜怎么落地：轻量产品形态、技术架构与 Loop 能力进化

Product Form

4.1｜产品形态：轻量嵌入式 SkillOps，而不是让用户迁移到新平台

1. 形态结论

用户在哪里写，我们就嵌入哪里；用户在哪里用，我们就记录哪里；用户在哪里改，我们就提示同步；用户在哪里重复造轮子，我们就推荐已有 Skill。

2. 整体产品架构

用户工作入口
Codex、Claude、WorkBuddy、内部 Agent 客户端、Agent 协作平台。
客户端轻量适配层
Codex Plugin、Claude Skill + Hook、WorkBuddy 原生入口、命令面板、右侧抽屉、保存为 Skill。
SkillOps MCP Server
提供查重、生成、质检、修复、注册、Fork、回源、使用事件和改进建议等统一能力接口。
SkillOps Core
管理 Skill 元数据、版本血缘、质量测试、使用事件、改进建议、个人与团队资产范围、市场连接器。
后台治理台
给团队负责人、市场运营者和管理员使用；不作为普通员工创建 Skill 的主入口。
分发连接层
连接内部 Skill 市场、团队仓库、客户私有 Registry 或未来外部平台。

3. 为什么不能只做成一个 Skill

Skill 可以作为最轻的用户入口，但不能承担完整系统职责：

  • 难以统一管理跨客户端版本和使用反馈。
  • 难以承担组织级 Registry、权限、审计和版本血缘。
  • 难以持续推送基于运行数据的改进建议。

4. 为什么不能只做成 Codex 插件

Codex 插件适合在 Codex 内分发入口，但会把产品锁定在单一客户端。集团内部还存在 WorkBuddy、Claude、Agent 协作平台和其他入口，因此：

插件、Skill、Hook 和原生入口都只是客户端适配器；产品本体必须是独立的 SkillOps Core 和统一能力接口。

5. 各客户端的推荐形态

客户端推荐形态用户体验
CodexPlugin + MCP通过命令或对话直接查重、创建、检查、Fork、提交和查看改进建议。
ClaudeSkill + Hook + MCP主动调用 SkillOps，或在保存、修改、任务结束等事件中自动提示。
WorkBuddy / 内部客户端原生入口“保存为 Skill”“查找类似 Skill”“检查当前 Skill”等不打断主流程的入口。
Skill 市场上架前工作台 / 连接器接收质量报告、相似检查、测试和适配信息，再进入正式审核。

6. 轻量组件的五个核心能力

Skill Lens

识别用户是否在反复执行同类任务、修改已有 Skill，或已经产生可沉淀能力。

Skill Search

创建前查询相似 Skill，建议复用、Fork 或新建。

Skill Linter

写的时候检查触发、输入、输出、边界、工具、测试和适配。

Registry Sync

管理 local、private、team、market 四种资产范围及版本血缘。

Improvement Agent

根据使用、失败、人工修改和派生版本，持续推送改进建议。

7. 使用数据采集原则

数据第一阶段建议
Skill ID、版本、客户端、触发方式采集
是否执行成功、是否人工修改、是否 Fork / 回源采集
用户评分与反馈标签采集
完整 Prompt、完整输出、业务敏感文件默认不采集
Loop Relationship

4.2｜与 Loop 的关系：Loop 负责把任务做完，SkillOps 负责让“怎么做”持续进化

1. 三个对象的边界

对象核心问题主要职责
Loop Runtime这次任务如何持续完成计划、执行、检查、重试、调度、恢复和退出。
Skill某一类任务应该怎么做封装方法、知识、步骤、工具规则、输入输出和检查标准。
SkillOps这些方法如何长期变好版本、测试、反馈、复用、合并、治理和组织级传播。

2. Loop 的三种“进化”层级

任务内纠错

Loop 在当前任务中执行、检查、发现问题并重新调整，直到达到完成标准。

单个 Loop 工程优化

开发者根据多次运行，修改流程、Prompt、Judge、工具顺序和重试条件。

组织能力进化

跨任务、跨 Loop、跨团队识别共性问题，把成功方法和失败经验沉淀为可共享 Skill。

3. SkillOps 是 Loop 的能力进化层

没有 SkillOps，Loop 仍然可以在一次任务里纠错；但改进容易停留在当前任务、当前项目或当前客户端，无法稳定传播为组织能力。

4. 没有 SkillOps 时，Loop 会遇到的问题

  • 每次运行都在重复学习，人工补充没有进入稳定 Skill。
  • A 团队修复的问题，B 团队不知道，多个 Loop 重复试错。
  • 无法判断失败究竟来自目标、工具、知识、Judge 还是 Skill。
  • 改进绑定在某个客户端或某个 Loop 中，无法跨场景复用。

5. SkillOps 如何具体帮助 Loop

  1. 执行前：为 Loop 推荐最匹配、最新稳定、已通过测试的 Skill 及其知识和工具依赖。
  2. 执行中：记录 Skill 版本、失败步骤、人工介入和工具调用表现。
  3. 执行后：从人工修改、重复失败和派生版本中提取改进信号。
  4. 跨 Loop：比较多个版本，识别通用改动和业务分支，推动合并或标准化。
  5. 反向沉淀：把成功 Loop 中稳定、可复用的部分抽取成新的 Skill。

6. 两个循环

任务执行循环

目标
→
Loop 执行
→
调用 Skill 与工具
→
检查与修正
→
完成

能力进化循环

汇总多个 Loop 结果
→
识别重复问题
→
更新并测试 Skill
→
发布新版本
→
重新进入 Loop
Loop 解决 Agent 会不会持续干活；SkillOps 解决企业能不能因为持续干活而持续变强。

7. 边界控制

SkillOps 可以管理 Loop 使用的能力，但不应变成 Loop 管理平台。

SkillOps 管理Agent 引擎 / Loop Runtime 管理
使用了哪个 Skill、哪个版本、表现如何、是否该改进任务当前进行到哪里、如何调度、何时重试、何时退出
能力如何复用、合并、测试和版本化上下文、长任务、计算资源、Agent 状态和失败恢复
Cross-team Collaboration

Cross-Department Collaboration

5.1｜与内部 Skill 市场的关系及跨部门依赖：业务上强协同，技术上松耦合

1. 依赖判断

SkillOps 未来会依赖 Skill 市场的正式审核与分发价值，但不能形成生存性依赖。市场应该是重要出口和连接器，而不是 SkillOps 的唯一入口、唯一数据库和唯一运行前提。

→

5. 分阶段依赖策略

第一阶段｜低依赖，先证明核心价值

在 Agent 客户端中完成创建、查重、质检、Fork、团队使用和改进建议；需要上架时导出标准 Skill 包，人工提交市场。 即使市场团队暂时不提供接口，SkillOps 仍能独立运行。

第二阶段｜中度依赖，建设发布连接器

读取已上架 Skill 元数据、查询相似 Skill、一键提交审核、同步审核状态和正式版本。

第三阶段｜深度协同，但保持数据边界

把 SkillOps 作为市场上架前质量工作台，在市场中展示质量、测试、版本血缘和维护状态；双方仍各自保存职责范围内的事实数据。

6. 跨部门真实风险与应对

风险应对方式
产品归属冲突明确市场管审核分发，SkillOps 管上游生产和下游运行反馈，不以“重做完整 Skill 平台”对外表达。
对方优先级不一致MVP 支持标准包导出、人工提交和状态回填，不等待市场 API 才启动。
无法获得市场全部数据先查 SkillOps 自身个人与团队库；市场目录采用离线同步或基础接口，逐步升级。
市场团队后续补同类功能聚焦其难以覆盖的 Agent 客户端入口、运行反馈、跨客户端版本和 Loop 能力进化闭环。

7. 健康依赖的判断标准

假设 Skill 市场团队未来六个月不提供开发资源，SkillOps 是否仍能完成查重、创建、质检、Fork、团队共享、使用反馈和改进建议？ 如果答案是“能”，依赖是健康的；如果不接市场就无法保存、使用和验证，说明架构已经被市场绑死。
最稳的合作关系是：业务上强协同，技术上松耦合，数据上有边界，产品上互补，组织上不互相替代。
Roadmap & Commercialization

06｜怎么推进：内部验证、组织级治理与对外 B 端产品化

Roadmap

6.1｜内部落地路线、验证指标与边界

第一阶段：人工与半自动验证

周期：1 周至 1 个月。

目标：验证 SkillOps 是否能减少重复创造、提高质量、促进复用和回源。

试点对象：研发、设计 / 文档、FDE / 交付、产品 / 业务团队。

先不开发完整系统，先形成：

  • Skill 需求澄清模板。
  • 标准结构与质量评分表。
  • 测试用例模板。
  • 多平台适配差异说明。
  • 5—10 个真实 Skill 的人工评审和改前改后对比。

第二阶段：轻量 MVP

推荐组合：1 个 SkillOps Core + 1 个 MCP Server + 1 个 WorkBuddy 原生入口 + 1 个 Codex Plugin。

优先级功能验证的问题
P0相似 Skill 搜索能否减少重复建设。
P0Skill 快速创建 / 从任务抽取能否降低创建门槛。
P0质量检查与测试样例能否提高稳定性和减少返工。
P0个人 / 团队保存能否在不进入市场的情况下形成可复用资产。
P0Fork 与版本血缘能否解决修改不同步。
P0使用元数据采集能否形成持续改进依据。
P1提交回源与合并建议能否促进团队共创。
P1市场上架前检查报告能否被现有市场采用。
P1改进建议推送能否推动 Skill 持续进化。

第三阶段：接入 Skill 市场与组织级治理

当内部价值被证明后，再建设：

  • 市场目录查询和一键提交审核。
  • 上架前质量门禁。
  • 团队能力资产库和版本血缘图。
  • 高价值 Skill 推荐、失效清理和维护提醒。
  • 跨客户端运行反馈和 Loop 能力进化分析。

验证指标

指标支持信号
Skill 创建效率创建和修改时间明显下降。
重复建设创建前能够发现并复用已有 Skill。
质量提升测试通过率、输出稳定性提高，返工和人工修正减少。
复用行为出现真实 Fork、团队共享和跨人员使用。
贡献回源用户修改后愿意提交原作者或团队主版本。
平台协同Skill 市场愿意将质量报告作为上架前流程的一部分。
持续使用至少一批内部作者和团队在真实任务中持续使用。

反证信号

反证说明
用户只想快速写，不愿意质检质量治理需求弱，需要收缩为个人创建与管理工具。
相似 Skill 推荐没人使用组织复用意愿弱，查重价值不足。
Fork 后无人回源团队贡献机制不成立，需要重新设计激励和流程。
业务 Skill 大多真的是一次性不适合重治理，应强化临时与个人模式。
产品被认为增加负担入口或治理过重，必须进一步嵌入和自动化。
无法证明任务结果改善价值停留在管理想象，不应继续做重。
External Commercialization

6.2｜对外 B 端 SaaS 与私有化方向：从内部验证走向企业能力资产治理

1. 对外定位

对外不应定位为“Skill 写作工具”或“新的 Skill 市场”，而应定位为企业 Agent 能力资产生产、治理和持续进化平台。

2. 目标客户

客户类型适配度原因
已经使用多个 Agent 工具的大中型企业高能力分散、平台碎片化、私有治理和复用需求强。
AI 转型中的交付、咨询、FDE 团队高需要把项目经验和专家方法转化为可复用资产。
具备内部开发者平台的科技公司中高具备 Skill 标准化、测试和版本管理需求。
中小企业和个人开发者较低治理阶段尚早，付费意愿和组织复用需求较弱。

3. 产品形态

版本主要能力目标客户
SaaS 轻量版创建、质检、版本、共享、反馈和改进建议AI 团队、交付团队、内部创新团队
企业版私有 Registry、权限、审计、测试、适配和生命周期治理大中型企业
私有化版接入企业 Agent 平台、IM、云盘、知识库、MCP、SSO 与内部市场集团、金融、政企等客户

4. 对外壁垒

单纯生成 Skill 容易被各平台原生能力覆盖，真正需要积累的壁垒是：

企业场景理解 质量评估体系 测试数据集 权限与审计 多客户端适配 版本血缘 Loop 运行反馈闭环
Decision & Ask

07｜最终行动判断与需要的协同支持

1. 最终判断

建议做。第一阶段以轻量嵌入方式验证内部价值，核心只抓“查重、创建、质检、Fork / 同步、使用反馈和改进建议”； 内部闭环成立后，再进入 Skill 市场深度协同、组织级治理和对外 B 端产品化。

2. 项目定位

该项目应定位为集团 AI 能力资产基础设施，而不是单点 Skill 工具；它向下连接知识、工具和任务，向上支撑超级个体、超级团队和超级组织，并具备未来 B 端 SaaS 与私有化能力。

3. 内部支持

试点场景授权

选择研发、FDE / 交付、设计 / 文档、产品 / 业务等真实团队做小规模试点。

现有系统协同

获得类 Codex、WorkBuddy、云盘 / 知识库、MCP 工具平台和 Skill 市场的基础规范与必要接口支持。

跨部门边界确认

明确 SkillOps 管生产、质量、运行反馈和进化；Skill 市场管安全审核、正式发布与分发。

4. 最后收束

AI 时代企业真正的壁垒，不只是拥有多少知识、多少 Agent 或多少模型调用，而是能否把知识、流程、工具、经验和复盘， 持续沉淀成可复用、可治理、可持续进化的组织能力。

如果没有 SkillOps，企业内部会出现大量分散的个人 Skill、重复建设的 Prompt、不可复用的项目经验和难以治理的 Agent 能力； Loop 可以持续完成任务，但学习和修正容易停留在局部。

有了 SkillOps，每一次高质量任务都可能成为下一次任务的基础，一个人的改进可以成为团队的标准，一个 Loop 的经验可以成为整个组织的能力。

SkillOps 的长期结果，是让集团从“使用 AI 工具”升级为“持续积累和进化 AI 时代的组织能力资产”。
报告核心结论： 业务上强协同，技术上松耦合；用户侧轻量嵌入，平台侧统一治理；先用内部真实场景验证产品价值，再逐步形成集团级 AI 能力资产基础设施与对外 B 端产品。
